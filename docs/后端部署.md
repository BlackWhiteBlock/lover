# Lover 后端自动化部署说明

> 与同机 Family 服务共存。部署路径、端口、PM2 进程名、数据库、`.env` 均隔离。

## 1. 与 Family 的隔离约定

| 项 | Family（参考） | Lover（本仓库默认） |
|----|----------------|---------------------|
| 代码目录 | `/opt/family/backend` | `/opt/lover/backend` |
| 监听端口 | `4000` | `4001` |
| PM2 名称 | `family-backend` | `lover-backend` |
| 数据库 | Family 自己的库 | **独立** `lover` 库 |
| 环境文件 | `/opt/family/backend/.env` | `/opt/lover/backend/.env` |
| 本地上传目录 | （各项目各自） | `/opt/lover/backend/data/uploads` |

Workflow 会拒绝：

- `BACKEND_DEPLOY_PATH` 落在 `/opt/family/**`
- `BACKEND_PM2_APP_NAME=family-backend`
- `BACKEND_DEPLOY_PORT=4000` 或服务器 `.env` 里 `PORT=4000`

部署不会 rsync 到 Family 目录，也不会 `pm2 restart family-backend`。

## 2. GitHub 配置

路径：仓库 → **Settings → Secrets and variables → Actions**

### Secrets

| Name | 说明 |
|------|------|
| `BACKEND_DEPLOY_HOST` | 服务器 IP 或域名（可与 Family 相同） |
| `BACKEND_DEPLOY_USER` | SSH 用户（可与 Family 相同） |
| `BACKEND_DEPLOY_SSH_KEY` | 部署用私钥全文 |
| `BACKEND_ENV_FILE` | （推荐）首次部署时写入服务器的完整 `.env` 内容 |

`BACKEND_ENV_FILE` **只在服务器尚无 `.env` 时**复制为 `.env`；之后每次发布**不会覆盖**已有 `.env`。

### Variables（可选，有默认值）

| Name | 默认 | 说明 |
|------|------|------|
| `BACKEND_DEPLOY_SSH_PORT` | `22` | SSH 端口 |
| `BACKEND_DEPLOY_PATH` | `/opt/lover/backend` | 部署目录 |
| `BACKEND_DEPLOY_PORT` | `4001` | 健康检查回退端口（真正端口以 `.env` 的 `PORT` 为准） |
| `BACKEND_PM2_APP_NAME` | `lover-backend` | PM2 进程名 |
| `BACKEND_NPM_REGISTRY` | `https://registry.npmmirror.com` | npm 镜像 |

SSH 密钥可与 Family 共用同一把；**务必使用独立 `BACKEND_ENV_FILE` / 服务器 `.env`**，不要复用 Family 的数据库或 JWT / 短信 / 七牛配置（除非你明确要共用存储且能接受隔离风险）。

## 3. 生产 `.env` 要点

在 Family 旁并行部署时，至少保证：

```env
NODE_ENV=production
HOST=0.0.0.0
PORT=4001
DATABASE_URL=postgres://...@localhost:5432/lover
PUBLIC_BASE_URL=https://api.your-lover-domain.com
TRUST_PROXY=true

# 生产禁止 SMS_PROVIDER=dev，强制七牛等（见 backend/.env.example 与 config 校验）
SMS_PROVIDER=aliyun
STORAGE_PROVIDER=qiniu
# ... JWT / 阿里云短信 / 七牛密钥使用独立强随机值
```

部署前在服务器（或宝塔）创建独立 PostgreSQL 库，例如在宝塔「数据库 → PostgreSQL」中新建库 `lover`。

要求 **PostgreSQL 14+**（迁移使用核心函数 `gen_random_uuid()`，**不依赖** `pgcrypto`。宝塔预装 PG 常缺少 contrib 扩展，本项目已避免该依赖）。

## 4. 宝塔反向代理

1. 新建网站 / 反向代理站点，域名指向 Lover API（与 Family 域名分离）。
2. 反向代理目标：`http://127.0.0.1:4001`（或与 `.env` 中 `PORT` 一致）。
3. 开启 HTTPS（Let’s Encrypt）。
4. 如需限制 Swagger，对 `/docs`、`/openapi.json` 做 IP 白名单或额外鉴权。

Family 站点继续指向 `4000`，互不影响。

## 5. 触发方式

- `main` 分支推送且变更落入 `backend/**` 或本 workflow 文件
- Actions 页手动 **Run workflow**

流水线：`validate`（typecheck + test + build）→ `deploy`（rsync → `npm ci` → `build` → `db:migrate` → PM2 restart → `/health`）。

未配置部署 Secrets 时，validate 仍会跑，deploy 会 skip 并提示，不会改动服务器。

## 6. 首次上线检查清单

- [ ] GitHub Secrets / Variables 已按上文配置
- [ ] 宝塔已开独立反向代理到 `4001`
- [ ] PostgreSQL 已建 `lover` 库，且 `DATABASE_URL` 正确
- [ ] `.env` 中 `PORT≠4000`，`PM2` 名为 `lover-backend`
- [ ] 手动触发 workflow 成功，`curl https://你的域名/health` 正常
- [ ] `pm2 list` 中同时存在 `family-backend` 与 `lover-backend`，且 Family 健康检查仍通

## 7. 回滚建议

```bash
cd /opt/lover/backend
# 如有备份目录可回拷代码后：
npm ci
npm run build
pm2 restart lover-backend --update-env
```

不要对 `family-backend` 执行 restart/delete。数据库迁移默认向前兼容；破坏性迁移需单独预案。
