## 1. 一句话结论

**Lover** 是「双人私密情侣空间」产品：Android 原生客户端 + Fastify 模块化单体后端。  
MVP **业务闭环已打通**（登录 → 绑定 → 天数/媒体/纪念日/信件 → 解绑/退出），但相对 Figma 视觉仍是「同色系功能原型」，且客户端尚未接齐后端已暴露的编辑/删除等能力。

后续开发应优先：**UI 对齐设计稿 → 补齐 CRUD 交互 → Phase 2 能力（装扮/愿望单/推送）**，而不是推倒重来。

## 3. 产品理解（对齐需求 + Figma）

### 3.1 定位

- 品牌：`lover.` / *Two Hearts One World*
- 不是社交 Feed，是 **两人专属世界**
- 对称成员权限（无性别角色、无亲子层级）

### 3.2 五 Tab 信息架构

| Tab | Figma | Android 文案 | 核心内容 |
|-----|-------|--------------|----------|
| Home | 空间 | 首页 | 品牌头、相爱天数、写情书/存瞬间、近期掠影 |
| Gallery | 时光 | 时光 | 照片/视频双列相册 |
| Anniversary | 纪念 | 纪念 | 里程碑 / 年度纪念 + 倒计时 |
| Letters | 信封 | 信封 | 即时信 + 时间胶囊 |
| Profile | 我们 | 我们 | 情侣资料、解绑、退出、设置入口 |

### 3.3 MVP 范围内 / 外

**范围内（已实现或基础设施齐）：** 短信登录、邀请绑定、天数、媒体、纪念日、信件胶囊、双方确认解绑、退出。

**范围外（后续）：** 空间装扮、共同愿望单、推送、年度报告、小组件、字段级加密、冷静期/注销导出。

### 3.4 Figma 相对实现的「真源」价值

`UI-Figma/src/app/App.tsx` + `LoverComponents.tsx` 定义了：

- 岛屿底部导航 + 三 Tab FAB
- 视觉语言：`#fffcfb`、rose/stone、大圆角、毛玻璃、衬线标题
- 各模块文案骨架与卡片形态（纪念日大图卡、信件邮票、居中 Milestone 等）

Figma **未画出** 登录页与绑定页；这些由 Android 实装补齐。

## 4. 总体架构

```text
Android (Compose)
  ViewModel → AppRepository → ApiService / TokenStore / AssetUploader
           │ HTTPS JSON
           ▼
Fastify 模块化单体
  auth | couples | assets | media | anniversaries | letters | bootstrap
           │
    ┌──────┼───────────┐
    ▼      ▼           ▼
PostgreSQL  存储 Provider   阿里云短信
            (local | 七牛私有)
```

设计原则：

1. **空间隔离**：一切业务数据经服务端解析的 `activeSpaceId` 过滤；客户端不能传他人 `spaceId` 越权。
2. **对称双人**：一用户最多一个 active 空间；一空间最多两名 active 成员。
3. **媒体私有**：三阶段上传；阅读走短期签名 URL；生产强制七牛私有 Bucket。
4. **胶囊以服务端时间为准**：未解锁响应不含 `content`/`summary`；客户端只信 `isUnlocked`。

## 5. 后端理解

### 5.1 技术栈与入口

| 项 | 值 |
|----|-----|
| 运行时 | Node.js 20+ |
| 框架 | Fastify 5 + TypeScript |
| 校验 | Zod（拒绝额外字段） |
| DB | PostgreSQL 14+ |
| 入口 | `backend/src/server.ts` → `buildApp()` |
| 默认端口 | `4000` |
| 文档 | `/docs`，`/openapi.json` |

常用脚本：`npm run dev` / `typecheck` / `test` / `build` / `db:migrate`。

### 5.2 模块职责

| 模块文件 | 职责 |
|----------|------|
| `modules/auth.ts` | 短信发送/登录、refresh、logout、`/api/me` |
| `modules/couples.ts` | 空间、邀请、解绑申请/确认/取消 |
| `modules/assets.ts` | 上传凭证、local 上传、complete、sign、私有下载 |
| `modules/media.ts` | 时光相册业务记录 CRUD |
| `modules/anniversaries.ts` | 纪念日 CRUD + 服务端 countdown |
| `modules/letters.ts` | 信件 CRUD + 胶囊正文剥离 |
| `modules/bootstrap.ts` | 首页聚合：空间 + 天数 + 最近 6 条媒体 |
| `domain.ts` | 纯领域：`lovingDays`、`anniversaryCountdown`、`canAcceptInvite`、`isLetterUnlocked` |
| `storage/*` | Local / Qiniu Provider、object key 安全 |

### 5.3 核心数据表

```text
users, sms_codes, auth_sessions
couple_spaces, couple_members, couple_invites, unbinding_requests
media_assets, media_items
anniversaries, letters
```

迁移：`001_initial.sql`、`002_qiniu_storage.sql`，经 `schema_migrations` + advisory lock。

关键约束：

- 一用户一 active 空间（partial unique index）
- 一空间最多两名 active（trigger）
- 信件：`instant` 无 `unlock_at`；`capsule` 必须有 `unlock_at`

### 5.4 关键业务规则（必须背住）

**相爱天数：** `lovingDays = max(0, 日历差天数 + 1)`，**含在一起当天**。  
以服务端 `bootstrap.lovingJourney.days` 为准（客户端 `DateUtils` 仅测试用）。

**时间胶囊：**

- 创建时 `unlockAt` 必须晚于当前时间
- 解锁判定：`unlockAt <= now`（读时判定，无定时任务）
- 未解锁：任何一方（含作者）都看不到 `content`/`summary`

**解绑：**

1. A 申请 → pending（同空间同时仅一条 pending）
2. B 确认 → 双方 membership inactive，空间 `dissolved`，邀请取消
3. 历史数据保留，但双方立即失去 active 访问权

**媒体上传：**

```text
token → 客户端直传 (local multipart | 七牛 form) → complete(校验真实对象) → 业务引用 ready asset → sign 阅读
```

- Object key 仅服务端生成：`couples/{spaceId}/{YYYY-MM}/{24hex}.{ext}`
- 生产禁止 `STORAGE_PROVIDER=local`、禁止开发短信与默认 JWT 密钥

### 5.5 API 速查

```text
POST /api/auth/sms/send|login
POST /api/auth/refresh|logout
GET  /api/me

GET|PATCH /api/couple-space
POST /api/couple-invites|accept
POST /api/couple-space/unbinding[/:id/confirm|cancel]

GET  /api/bootstrap
POST /api/media-assets/token|complete|/:id/sign
GET|POST|PATCH|DELETE /api/media
GET|POST|PATCH|DELETE /api/anniversaries
GET|POST|PUT|DELETE /api/letters
```

错误信封：`{ error: { code, message, details? } }`。

### 5.6 后端成熟度与缺口

| 已扎实 | 缺口 / 注意 |
|--------|-------------|
| 鉴权会话、空间隔离、胶囊剥离 | OpenAPI 大量路径为 stub |
| 双存储 Provider、路径越狱防护 | 无完整 DB E2E，多为 unit + inject |
| 限流、生产 fail-fast | 限流内存单实例；多副本需 Redis |
| 领域纯函数单测 | 删 media_item 不清理 blob；无用户资料 PATCH |

## 6. Android 客户端理解

### 6.1 技术栈

| 项 | 值 |
|----|-----|
| 语言 | Kotlin, JDK 21 |
| UI | Jetpack Compose + Material 3 |
| DI | Hilt |
| 网络 | Retrofit + OkHttp + Kotlin Serialization |
| 存储 | DataStore Preferences |
| 图片/视频 | Coil / Media3 ExoPlayer |
| SDK | min 26，compile/target 36 |
| 包名 | `com.lover.app`（debug: `com.lover.app.debug`） |
| API | `BuildConfig.API_BASE_URL`，默认模拟器 `http://10.0.2.2:4000/` |

### 6.2 分层与会话门控

```text
MainActivity
  ├─ 无 token → AuthScreen
  ├─ 有 token 无 couple → CoupleScreen
  └─ 已绑定 → MainScreen（五 Tab）
         ↓
    MainViewModel / AuthViewModel / CoupleViewModel
         ↓
    AppRepository (@Singleton)
         ├── ApiService
         ├── TokenStore
         ├── AssetUploader（独立客户端，永不带 Lover Token）
         └── Media Resolver / Thumbnail
```

**没有 Navigation-Compose NavHost**；根层是显式状态机，Main 内用 `MainTab` + `AnimatedContent`。

### 6.3 包结构（改代码入口）

```text
com.lover.app/
├── MainActivity.kt / LoverApplication.kt
├── core/
│   ├── data/      AppRepository, TokenStore
│   ├── design/    LoverTheme
│   ├── di/        AppModule
│   ├── media/     流式上传、封面抽取
│   ├── model/     Models.kt（契约 DTO）
│   ├── network/   ApiService, Interceptor, Authenticator, Uploader
│   └── util/      DateUtils（仅测试路径）
└── feature/
    ├── auth/
    ├── couple/
    └── main/      MainScreen + MainViewModel（体量最大）
```

### 6.4 会话与缓存

- DataStore key：`app_state_v2`，整份 `AppState`（tokens + 业务缓存）
- Access 同源注入；第三方 host **剥离** Authorization
- 401 → Authenticator 单飞 refresh，失败清会话
- 冷启动：先读缓存 → `/api/me` → `refreshAll()`（bootstrap + 各列表 + sign）

### 6.5 媒体上传管线（客户端）

```text
Photo Picker
  → resolve（图 ≤30MB / 视频 ≤200MB，流式 RequestBody）
  → 视频抽 JPEG 封面
  → uploadAsset：token → 直传 → complete（视频再传封面）
  → POST /api/media
  → refreshAll + signAsset 补 URL
```

注意：服务端 MIME 上限更宽（单文件可达 300MB），客户端更严。

### 6.6 UI 相对 Figma 的现状

**已对齐：** 五 Tab 结构、岛屿导航思路、rose 色系、`#FFFCFB`、衬线标题、文案骨架、FAB 出现规则、首页快捷入口。

**明显差距：** 毛玻璃与光斑动效、居中 Milestone 大卡、纪念日大图封面卡、信件邮票/标签样式、Home Tab 名「首页」≠「空间」、愿望单等 Profile 子功能空壳。

定位：**功能可演示的 MVP**，不是像素级设计还原。

### 6.7 客户端对后端能力的使用缺口

| 后端已有 | Android 现状 |
|----------|--------------|
| Media PATCH/DELETE | 未接 UI |
| Anniversary PATCH/DELETE、封面 `coverAssetId` | 未接 |
| Letter PUT/DELETE | 未接 |
| `PATCH /api/couple-space` | 未接 |
| 取消邀请 | 未接 |
| media cursor 分页 | 固定 `limit=100` |
| `needsTogetherDate` | 未驱动引导 UI |

## 7. 端到端主路径

### 7.1 登录

```text
发短信 → 验证码哈希入库 → 登录消费验证码
  → 签发 access + refresh（DB 存 refresh SHA-256）
  → Android 持久化 → me → 有空间则 refreshAll
```

开发短信固定码默认 `123456`，响应可带 `devCode`。

### 7.2 绑定

```text
A: POST /couple-invites（可顺带创建空间）→ 明文邀请码仅此时返回
B: accept(code) → 事务 + 行锁 + 容量校验
→ 双方进入 Main
```

### 7.3 首页

`GET /api/bootstrap` → space + lovingJourney.days + recentMedia(6)。

### 7.4 存瞬间 / 写信 / 加纪念日

均先走各自创建 API，再客户端刷新列表。  
胶囊日期：客户端把所选本地日历日转为「本地午夜 + offset」的 ISO，服务端存 `unlock_at`。

### 7.5 解绑 / 退出

解绑需双方确认；退出调 logout 并清本地会话。

## 8. 安全与运维要点

- 生产：阿里云短信 + 七牛私有 + 强随机密钥 + HTTPS
- Release Android：禁止明文 HTTP，`LOVER_API_BASE_URL` 注入
- 写入限流：短信 / 登录 / 邀请 / 上传 token
- `/docs` 生产应由反向代理限制访问
- CI（`.github/workflows/ci.yml`）：
  - Backend：`npm ci` → typecheck → test → build
  - Android：`testDebugUnitTest` → `lintDebug` → `assembleDebug`，上传 debug APK

## 12. 给后续会话的上下文约定

作为本项目全栈工程师时，默认遵守：

1. **视觉以 `UI-Figma` 为准，行为以 `backend` 领域规则为准，Android 跟随后端契约。**
2. 不引入微服务拆分；不把 `UI-Figma` 当正式前端交付。
3. 改协议时同步维护 Android `ApiService`/`Models` 与相关单测。
4. 涉及日期/胶囊/天数的逻辑只信服务端，避免客户端另算一套业务真相。
5. 生产安全红线：不清提交密钥；不放宽越权；不在客户端暴露锁定胶囊正文。
