# 阿里云本机号一键登录（PNVS）接入说明

> 面向技术同学的参考文档：说明 Lover 项目如何接入阿里云号码认证「一键登录」，覆盖产品选型、端到端链路、双端（Android / HarmonyOS）与后端实现要点，以及踩坑记录。  
> 官方产品名：**号码认证服务（PNVS）** · 控制台：https://dypns.console.aliyun.com/

---

## 1. 我们选了什么能力

阿里云号码认证常见有两类能力，容易混淆：

| 能力 | 用户体验 | 服务端接口 | 我们是否采用 |
|------|----------|------------|--------------|
| **一键登录** | 拉起运营商授权页 → 用户确认 → 拿到 `loginToken` → 服务端换手机号并建会话 | `GetMobile` | ✅ 采用 |
| 本机号码校验 | 用户仍输入手机号，SDK 校验「本机是否该号」 | 另一套 Verify 接口 | ❌ 未采用 |

**产品交互定稿：**

- 登录页以「本机号码一键登录」为主按钮；
- 「其他手机号 / 验证码登录」为短信兜底；
- 一键登录失败或不支持时，自动展开短信登录，不阻断注册/登录。

## 2. 整体架构

```text
┌─────────────────┐     ┌─────────────────┐
│  Android App    │     │  Harmony App    │
│  Aliyun Auth SDK│     │  numberauth HAR │
└────────┬────────┘     └────────┬────────┘
         │ 1) GET /api/auth/pnvs/sdk-info?platform=android|harmony
         │    ← secretInfo + 协议页 URL
         │ 2) SDK 拉起授权页，用户同意
         │ 3) SDK 回调 loginToken（短时凭证，不是手机号）
         │ 4) POST /api/auth/pnvs/login { loginToken }
         ▼
┌────────────────────────────────────────┐
│  Lover Backend (Fastify)               │
│  ① 用 AK 调阿里云 GetMobile(loginToken)│
│  ② 得到真实手机号                      │
│  ③ 走与短信登录相同的 issueAuthSession │
│     → JWT / refresh / 新用户判定       │
└────────────────────────────────────────┘
```

**安全原则：**

- 客户端只拿「方案密钥（SDK Secret）」做 SDK 初始化，**不能**用它换手机号；
- 手机号只能由服务端用 RAM AccessKey 调 `GetMobile` 换取；
- 方案密钥通过 `sdk-info` 下发，避免写死在客户端二进制里（仍建议按环境区分 debug/release 方案）。

## 3. 控制台侧准备

### 3.1 创建认证方案

在号码认证控制台为每个平台/包名创建「认证方案」，拿到 **方案密钥（Secret）**。

**Android（开发包）：**

| 项 | 说明 |
|----|------|
| 包名 | Debug 构建为 `com.lover.app` + `applicationIdSuffix=.debug` → **`com.lover.app.debug`** |
| 签名 | Debug 使用本机 `~/.android/debug.keystore` 的 **MD5（无冒号、小写）** |
| Release | 包名 `com.lover.app` + release 证书 MD5，建议另建方案 |

获取 Debug MD5 示例（Windows）：

```powershell
keytool -exportcert -alias androiddebugkey `
  -keystore "$env:USERPROFILE\.android\debug.keystore" `
  -storepass android -file $env:TEMP\debug.cer
# 再对 debug.cer 做 MD5，去掉冒号并转小写
```

**HarmonyOS：**

| 项 | 说明 |
|----|------|
| 包名 | 与 `module` / `app` 中配置一致（本项目为 `com.lover.app`） |
| 指纹 | 使用当前签名证书指纹（注意：debug 自动签名与正式证书不同，需与控制台一致） |
| AppId | 号码认证侧认的是证书里的 **app-identifier / signatureInfo.appIdentifier**，不要误填 AGC 应用 ID |

### 3.2 RAM 权限

服务端 `GetMobile` 复用短信同一套 `ALIYUN_SMS_ACCESS_KEY_ID` / `SECRET`，对应 RAM 用户需具备号码认证取号权限，例如：

- `dypns:GetMobile`（或官方推荐的号码认证相关策略）

仅配了短信权限、没有 PNVS 取号权限时，客户端能出授权页，但服务端换号会失败。

### 3.3 SDK 下载

- Android：控制台下载，解压后得到 **3 个 aar**（auth / logger / main）
- Harmony：下载 `.har`，放入工程并按模块依赖声明

官方文档入口：

- Android 一键登录接入：控制台文档「Android 客户端接入」
- Harmony 一键登录接入：控制台文档「Harmony 客户端接入」

## 4. 后端实现

### 4.1 环境变量

见 `backend/.env.example`：

```env
PNVS_ENABLED=true
PNVS_ANDROID_SDK_SECRET=...   # 控制台 Android 方案密钥（整段 JSON/字符串原样）
PNVS_HARMONY_SDK_SECRET=...   # 控制台 Harmony 方案密钥
ALIYUN_SMS_ACCESS_KEY_ID=...
ALIYUN_SMS_ACCESS_KEY_SECRET=...
PUBLIC_BASE_URL=https://api.example.com   # 用于拼协议页绝对 URL
```

`PNVS_ENABLED=true` 时，校验至少配置一端 Secret，且必须有阿里云 AK（用于 GetMobile）。

### 4.2 接口

#### `GET /api/auth/pnvs/sdk-info?platform=android|harmony`

客户端启动登录页时调用，判断是否展示一键登录并初始化 SDK。

返回示例：

```json
{
  "enabled": true,
  "platform": "harmony",
  "secretInfo": "<方案密钥>",
  "privacyUrl": "https://api.example.com/legal/privacy",
  "termsUrl": "https://api.example.com/legal/terms"
}
```

`enabled` 为 `true` 的条件：

1. `PNVS_ENABLED=true`
2. 对应 platform 的 Secret 非空

`enabled=false` 时 `secretInfo` 为空字符串，客户端应只走短信。

#### `POST /api/auth/pnvs/login`

```json
{ "loginToken": "<SDK 回调的 token>", "nickname": null }
```

流程：

1. 校验 PNVS 已开启、AK 已配置；
2. 调用阿里云 `GetMobile`（SDK：`@alicloud/dypnsapi20170525`，endpoint `dypnsapi.aliyuncs.com`）；
3. 规范化手机号；
4. **复用短信登录同一套** `issueAuthSession`（创建/查找用户、发 JWT、新用户返回 201）。

实现位置：`backend/src/modules/auth.ts`（`fetchMobileFromPnvs` + 路由注册）。

### 4.3 协议占位页

授权页需要可点击的隐私政策 / 用户协议 URL。本项目提供：

- `GET /legal/privacy`
- `GET /legal/terms`

静态 HTML 位于 `backend/legal/`。上线前应替换为正式法务文案。

## 5. Android 实现要点

### 5.1 依赖

将 3 个 aar 放入：

```text
android/app/libs/
  auth_*.aar
  logger_*.aar
  main_*.aar
```

`build.gradle.kts`：

```kotlin
implementation(fileTree(mapOf("dir" to "libs", "include" to listOf("*.aar"))))
```

说明见 `android/app/libs/README.md`。

未放入 aar 时：工程仍可编译；通过反射探测 SDK 类，探测失败则一键登录不可用、展示短信。

### 5.2 关键文件

| 文件 | 职责 |
|------|------|
| `core/auth/PnvsLoginHelper.kt` | 反射桥接 `PhoneNumberAuthHelper`（setAuthSDKInfo / checkEnv / getLoginToken / UI 配置） |
| `feature/auth/AuthViewModel.kt` | 拉 sdk-info → prepare → 一键登录 → `/pnvs/login` |
| `feature/auth/AuthScreen.kt` | 一键登录主按钮 + 短信兜底 UI |
| `AndroidManifest.xml` | **必须**注册授权相关 Activity |

### 5.3 Manifest（高危踩坑）

aar 自带 Manifest **不会**声明授权页 Activity，必须在 App Manifest 手写注册，否则常见 **600002「唤起授权页失败」**（底层多为 `ActivityNotFoundException: LoginAuthActivity`）：

```xml
<activity
    android:name="com.mobile.auth.gatewayauth.LoginAuthActivity"
    android:configChanges="orientation|keyboardHidden|screenSize"
    android:exported="false"
    android:launchMode="singleTop"
    android:screenOrientation="behind" />
<!-- 另需 AuthWebVeiwActivity、PrivacyDialogActivity，见工程 Manifest -->
```

权限至少：`INTERNET`、`ACCESS_NETWORK_STATE`、`ACCESS_WIFI_STATE`、`CHANGE_NETWORK_STATE`。

### 5.4 调用顺序（推荐）

1. `GET sdk-info?platform=android`
2. `PhoneNumberAuthHelper.getInstance` + `setAuthSDKInfo(secret)`
3. `setAuthUIConfig`（协议文案与 URL）
4. `checkEnvAvailable(SERVICE_TYPE_LOGIN)`（可选加速 `accelerateLoginPage`）
5. 用户点击 → `getLoginToken(activity, timeout)`
6. 回调 `600001`：授权页已拉起；`600000`：拿到 token → 调后端 `/pnvs/login`
7. 成功后 `quitLoginPage()`

本项目用反射封装，便于「无 aar 也能编过」；有 aar 后行为与官方 Demo 一致。

### 5.5 包名注意

| BuildType | applicationId |
|-----------|----------------|
| debug | `com.lover.app.debug` |
| release | `com.lover.app` |

控制台方案、Secret、安装包三者必须一致。用 debug 包测时，Secret 必须是 debug 方案的密钥。

## 6. HarmonyOS 实现要点

### 6.1 依赖

```text
harmony/entry/libs/numberauth_standard.har
```

`oh-package.json5`：

```json5
"dependencies": {
  "numberauth_standard": "file:./libs/numberauth_standard.har"
}
```

放入后在 `harmony/entry` 执行 `ohpm install`。说明见 `harmony/entry/libs/README.md`。

工程级 `harmony/build-profile.json5` 需开启：

```json5
products[].buildOption.strictMode.useNormalizedOHMUrl: true
```

否则接入字节码 HAR（`numberauth_standard` 及其运营商依赖）会报 `00306046`。

### 6.2 与 Android 的差异（重要）

鸿蒙 **不需要** 像 Android 那样在 `module.json5` 再注册一个「授权页 Ability」。

官方要求主要是：

1. 依赖 HAR；
2. 权限：`ohos.permission.INTERNET` / `GET_NETWORK_INFO` / `SET_NETWORK_INFO`；
3. 代码：`setAuthSDKInfo` → `setAuthConfig` → `getLoginToken`。

授权 UI 由 SDK 内部拉起（可通过 `AuthUiConfig` 定制，含协议链接等）。

**UI 配置注意：** 鸿蒙必须显式设置掩码/按钮的 `margin` 与 `AlignRuleOption`（参考官方 NumberAuthDemo）。只设协议 URL、不设布局时，常见异常为：按钮贴顶、掩码空白、复选框消失。自定义协议文案不要自带 `《》`，否则与 `vendorPrivacyPrefix/Suffix` 叠成 `《《隐私政策》》`。

### 6.3 关键文件

| 文件 | 职责 |
|------|------|
| `core/auth/PnvsAuthHelper.ets` | 封装 `PhoneNumberAuthHelper` |
| `pages/AuthPage.ets` | 与 Android 对齐的登录交互 |
| `module.json5` | 权限 |

平台参数：`sdk-info?platform=harmony`。

## 7. 客户端统一 UX 约定

双端登录页行为应对齐：

1. 进入页面即请求 `sdk-info` 并尝试 `prepare`；
2. 就绪：主按钮「本机号码一键登录」，次级入口展开短信；
3. 未就绪（未开开关 / 无 Secret / 无 SDK / 环境检测失败）：主按钮禁用或提示不可用，并展开短信；
4. 一键失败：提示错误（可带阿里云错误码），展开短信，不踢出登录页。

## 9. 常见错误码与排查

| 现象 / 码 | 常见原因 | 处理 |
|-----------|----------|------|
| 600002 唤起授权页失败 | Android 未注册 `LoginAuthActivity`；或网络/预取号失败 | 先查 Manifest；再查蜂窝网、重置飞行模式 |
| 600017 | 包名 / 签名 / Secret 不匹配 | 核对控制台方案与当前安装包 |
| sdk-info `enabled:false` | 未开 `PNVS_ENABLED` 或缺对应 Secret | 查 `.env` 与部署环境 |
| 授权页能开，登录失败 | 服务端 GetMobile / AK 权限问题 | 查后端日志与 RAM 策略 |
| 仅短信可见 | 未放 SDK 包；或 prepare 失败 | 查 libs / ohpm / 日志 |

Android Logcat 可过滤：`PnvsLogin`、`AuthPnvs`、`AuthSDK`。

## 11. 设计取舍（给二次开发参考）

1. **一键登录为主 + 短信兜底**：覆盖无 SIM、双卡异常、漫游、方案未配齐等场景。  
2. **方案密钥服务端下发**：改密钥不用发版；注意传输走 HTTPS，且 Secret 仍属敏感配置。  
3. **Android 反射桥接**：无 aar 时 CI/其他同学仍能编过；有 aar 后走真实 SDK。鸿蒙则直接依赖 HAR（缺文件时模块无法解析依赖）。  
4. **会话签发与短信共用**：避免「一键登录用户」与「短信用户」两套账户体系。  
5. **双端 UX 对齐**：同一产品话术与主次按钮结构，降低运营与测试成本。
